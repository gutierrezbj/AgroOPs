import { Pool } from "pg";
import bcrypt from "bcryptjs";
const pool = new Pool({ connectionString: process.env.DATABASE_URL });

await pool.query("UPDATE users SET password_hash=$1 WHERE email='juancho@systemrapid.io'", [hash]);

await pool.end();

  -e DATABASE_URL="postgres://agroops:${PG_PASS}@postgres:5432/agroops" \

rm _reset.mjs
echo "🔐 Nuevo password admin: ${NEW}"
REMOTE

## 3. Cierre del deploy (un comando cuando DNS verde)

### 3.1 Verificar DNS

```bash
# Bypassa cache: pregunta directo al NS autoritativo de Hostinger
dig @athena.dns-parking.com agroops.agrom.es +short
# debe devolver: 187.77.71.102
```

Si devuelve `72.62.41.234` (Servidor 1, IP viejo) o nada → Hostinger no propagó. Acciones:
1. Abrí panel Hostinger → Domains → agrom.es → DNS → verificá que el A record `agroops → 187.77.71.102` sigue ahí.
2. Si está, esperá más (puede tardar hasta 24h en casos malos).
3. Si no está, agregalo de nuevo + Save, esperá 15-60min.

### 3.2 Emitir certificado

```bash
ssh root@srs-staging \
  "certbot --nginx -d agroops.agrom.es \
   --non-interactive --agree-tos \
   --email gutierrezbj@gmail.com --redirect"
```

certbot va a:
1. Parsear `/etc/nginx/sites-available/agroops.agrom.es`
2. Validar dominio vía HTTP-01 challenge (necesita DNS → IP correcta)
3. Modificar el archivo: añadir bloque `:443 ssl` + cert paths + redirect 80→443
4. `nginx -s reload` automático
5. Schedule renewal automático (`/etc/cron.d/certbot` ya existe)

### 3.3 Verificación final

```bash
# Headers HTTPS
curl -sSI https://agroops.agrom.es/api/health | head -3
# debe responder HTTP/2 200

# JSON health
curl -sS https://agroops.agrom.es/api/health | jq '.'
# debe responder status=ok|degraded con DB+Redis en ok
```

Si todo OK → abrir `https://agroops.agrom.es/login` en navegador, login con credenciales admin.

## 7. Gotchas que aprendimos hoy (resumen rápido, detalle en lessons.md)

1. **`POSTGRES_PASSWORD` hex, no base64.** El compose interpola en `DATABASE_URL=postgresql://agroops:${X}@...` y `pg` parsea con `new URL()`. `/` o `+` rompen silenciosamente con error enmascarado.
2. **Middleware Next 16 deny-by-default.** El matcher `/((?!api/auth|...).*)` debe excluir TODA ruta API pública, incluida `api/health`. Sin esto, Docker HEALTHCHECK queda eterno en `starting`.
3. **Drizzle: lazy init via Proxy obligatorio.** Sin lazy init, `next build` falla durante "collect page data" porque importa `@/db` sin `DATABASE_URL`.
4. **`drizzle-kit migrate` CLI cuelga en non-TTY.** Bypass con la API programática `migrate()` de `drizzle-orm/node-postgres/migrator`.
5. **Hostinger DNS panel → NS autoritativos puede tardar >30min.** Diagnóstico: `dig @athena.dns-parking.com <dominio>` (no usa cache). No bloquear el resto del deploy en eso.
6. **SSH al VPS: usa Tailscale, no IP pública.** `root@srs-staging` (= 100.110.52.22). El IP público tiene fail2ban activo y se puede banear con 4-5 intentos fallidos.
